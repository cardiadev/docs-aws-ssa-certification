## El Mapa de Certificaciones AWS

AWS organiza sus certificaciones en **cuatro niveles de dificultad**. El SAA es el paso natural después de Cloud Practitioner y el examen Associate **más popular** de todo el ecosistema AWS:

| Nivel | Certificaciones | Experiencia recomendada | Color |
|---|---|---|---|
| **Foundational** | Cloud Practitioner | No requiere experiencia previa | ⬛ Negro |
| **Associate** | **Solutions Architect (SAA)** ← estás aquí, Developer, SysOps | 1 año en el rol | 🔵 Azul |
| **Professional** | Solutions Architect Pro, DevOps Engineer | 2-3 años en el rol | 🟢 Verde |
| **Specialty** | Security, Data Analytics*, ML, Networking, etc. | 3-5 años + deep dive | 🟣 Púrpura |

<Callout type="info" title="Ruta hacia DevOps Engineer Professional">
Si planeas obtener la certificación **DevOps Engineer Professional**, necesitas primero aprobar **Developer Associate** y/o **SysOps Associate**. Ambas alimentan esa certificación profesional.
</Callout>

### ¿Por qué el SAA específicamente?

- Es la certificación **más tomada** en todo AWS después de Cloud Practitioner
- Te da la **exposición más amplia** a servicios core de AWS — útil sin importar si vas a ML, Security o DevOps
- **No requiere** experiencia en codificación ni en operaciones — solo comprensión de servicios y diseño
- Todo el examen gira alrededor del **Well-Architected Framework** y sus 4 pilares

## ¿Qué es un Solutions Architect?

Un Solutions Architect **diseña soluciones técnicas basadas en necesidades de negocio**, típicamente conectando varios servicios de AWS para resolver un problema. Se ubica en un punto medio entre la visión estratégica y la implementación técnica:

| Tipo de Arquitecto | Nivel | Enfoque | Ejemplo de trabajo |
|---|---|---|---|
| **Enterprise Architect** | Estratégico | Gobernanza, políticas, big picture de 1000+ apps | Definir estándares de aprobación de software para toda la organización |
| **Solutions Architect** | Táctico | Multi-servicio, puente negocio ↔ técnica | Diseñar una arquitectura que conecte EC2 + RDS + S3 + CloudFront para un e-commerce |
| **Application/Software Architect** | Especialista | Un solo sistema o app en profundidad | Optimizar la arquitectura interna de una API en Python |

<Callout type="idea" title="Punto clave">
El SA es **más técnico** que el Enterprise Architect pero **más amplio** que el Application Architect. Por eso el examen SAA cubre tantos servicios — refleja la amplitud real del rol.
</Callout>

## ¿Quién debería tomar el SAA?

Sin importar tu área de especialización futura, el SAA te da una base sólida:

| Tu área de interés | ¿El SAA te sirve? | Razón |
|---|---|---|
| DevOps / SRE | ✅ Sí | Base para entender los servicios que vas a automatizar |
| Machine Learning | ✅ Sí | Necesitas entender la infraestructura donde correrán tus modelos |
| Cybersecurity | ✅ Sí | 30% del examen es seguridad — es la base de AWS Security Specialty |
| Desarrollo | ✅ Sí | Entenderás cómo se conectan los servicios que consume tu código |
| Gestión / Liderazgo | ✅ Sí | Te da vocabulario técnico para tomar decisiones informadas |

## Formato del Examen

| Aspecto | Detalle |
|---|---|
| Código | **SAA-C03** |
| Preguntas | 65 total (~130 minutos) |
| Tipo | Escenarios multi-servicio (no memorización pura) |
| Selección | Opción múltiple (1 correcta) o respuesta múltiple ("seleccione 2") |
| Aprobación | **720 / 1000** |

<Callout type="warn" title="Diferencia clave con Cloud Practitioner">
El CCP pregunta **qué** es un servicio. El SAA pregunta **cómo** combinar servicios para resolver un escenario. Las preguntas son de **análisis y aplicación**, no de recuerdo.
</Callout>

## Ejemplos de Preguntas del Examen

Estos ejemplos vienen directamente del material del curso y reflejan el estilo real de las preguntas SAA-C03.

### Ejemplo 1: NAT Gateway + Subnets Privadas

> Una empresa ejecuta una aplicación web de 3 capas en un VPC con múltiples AZs. Las instancias EC2 del tier de aplicación están en **subnets privadas** y necesitan descargar parches de internet. Sin embargo, las instancias **no pueden ser accesibles directamente desde internet**. ¿Qué acciones tomar? *(Seleccione 2)*

| Opción | ¿Correcta? |
|---|---|
| A) Configurar un NAT Gateway en una **subnet pública** | ✅ |
| B) Definir una route table personalizada con ruta al NAT Gateway y asociarla a las subnets privadas | ✅ |
| C) Asignar una Elastic IP a cada instancia | ❌ |
| D) Configurar un NAT Instance en una **subnet privada** | ❌ |
| E) Definir una route table personalizada sin asociarla | ❌ |

<Callout type="success" title="Razonamiento">
- Los **NAT Gateways deben vivir en una subnet pública** (si no, no tienen acceso a internet para hacer proxy)
- Las subnets privadas necesitan una **route table que apunte al NAT Gateway** para tráfico hacia internet
- Una Elastic IP en la instancia la expondría directamente — viola el requisito
- Un NAT Instance en subnet privada no funcionaría — necesita estar en subnet pública
</Callout>

### Ejemplo 2: Hibernación de EC2

> Un arquitecto quiere ahorrar costos en instancias EC2 que no necesitan correr durante un shutdown de 2 semanas. Las aplicaciones almacenan datos en **memoria de instancia** que deben estar presentes al reanudar. ¿Qué solución usar?

| Opción | ¿Correcta? |
|---|---|
| A) Tomar un snapshot de los volúmenes EBS y restaurar | ❌ |
| B) Parar las instancias y depender del almacenamiento EBS | ❌ |
| C) Ejecutar las apps con **hibernación habilitada** y hibernar antes del shutdown | ✅ |
| D) Anotar los tipos de instancia y relanzar con la misma configuración | ❌ |

<Callout type="success" title="Razonamiento">
La **hibernación** es la **única característica de EC2** que preserva la memoria de instancia (RAM) al disco. Al reanudar, la RAM se restaura exactamente como estaba. Parar la instancia (stop) solo preserva el volumen EBS, no la RAM.
</Callout>

### Ejemplo 3: Failover con ENI

> Una app de monitoreo en EC2 usa su **dirección IP privada** para recibir conexiones. Se necesita redirigir tráfico a una instancia standby si la principal falla. ¿Cómo diseñar la solución?

| Opción | ¿Correcta? |
|---|---|
| A) Desplegar un ALB y registrar ambas instancias | ❌ |
| B) Configurar DHCP para reasignar la misma IP | ❌ |
| C) Adjuntar una **ENI secundaria** con la IP privada; moverla al standby si falla la primaria | ✅ |
| D) Asociar una Elastic IP y migrarla entre instancias | ❌ |

<Callout type="success" title="Razonamiento">
- Un **ALB** es para configuraciones **activo-activo**, no activo-pasivo
- **DHCP** en AWS no permite reasignar IPs privadas de esta forma
- Una **ENI secundaria** con la IP privada **se puede mover** entre instancias — es la solución correcta para failover activo-pasivo con IP privada
- Una Elastic IP es pública — el escenario usa IP **privada**
</Callout>

### Ejemplo 4: Pre-signed URL en S3

> Una empresa de analytics quiere que una página web ejecute un script JavaScript que haga **GET requests autenticadas** al bucket S3 de la empresa. ¿Qué debe hacer el arquitecto?

| Opción | Análisis |
|---|---|
| Proveer a los usuarios con una **Pre-signed URL** para el script | ✅ — La pre-signed URL incluye un token de autenticación temporal que permite el GET request sin exponer credenciales |
| Habilitar CORS en el bucket | Necesario pero insuficiente — no resuelve la autenticación |
| Crear un usuario IAM público | ❌ — Nunca se exponen credenciales IAM en JavaScript del lado del cliente |

<Callout type="success" title="Razonamiento">
Una **Pre-signed URL** contiene un token con fecha de expiración que autoriza el request sin requerir que el usuario tenga credenciales de AWS. Es el mecanismo estándar para dar acceso temporal y autenticado a objetos en S3.
</Callout>
