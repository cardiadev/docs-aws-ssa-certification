## The AWS Certification Roadmap

AWS organizes its certifications into **four difficulty levels**. The SAA is the natural step after Cloud Practitioner and the **most popular** Associate-level exam in the entire AWS ecosystem:

| Level | Certifications | Recommended experience | Badge |
|---|---|---|---|
| **Foundational** | Cloud Practitioner | No prior experience required | ⬛ Black |
| **Associate** | **Solutions Architect (SAA)** ← you are here, Developer, SysOps | 1 year in role | 🔵 Blue |
| **Professional** | Solutions Architect Pro, DevOps Engineer | 2-3 years in role | 🟢 Green |
| **Specialty** | Security, Data Analytics*, ML, Networking, etc. | 3-5 years + deep dive | 🟣 Purple |

<Callout type="info" title="Path to DevOps Engineer Professional">
If you plan to pursue the **DevOps Engineer Professional** certification, you first need to pass **Developer Associate** and/or **SysOps Associate**. Both feed into that professional cert.
</Callout>

### Why the SAA specifically?

- It is the **most taken** certification in all of AWS after Cloud Practitioner
- Gives you the **broadest exposure** to core AWS services — useful regardless of whether you're heading to ML, Security, or DevOps
- **Does not require** coding or operational experience — only understanding of services and design
- The entire exam revolves around the **Well-Architected Framework** and its 4 pillars

## What Is a Solutions Architect?

A Solutions Architect **designs technical solutions based on business requirements**, typically connecting several AWS services to solve a problem. They sit at a midpoint between strategic vision and technical implementation:

| Architect Type | Level | Focus | Example Work |
|---|---|---|---|
| **Enterprise Architect** | Strategic | Governance, policies, big picture across 1000+ apps | Define software approval standards for the entire organization |
| **Solutions Architect** | Tactical | Multi-service, bridge business ↔ technical | Design an architecture connecting EC2 + RDS + S3 + CloudFront for an e-commerce platform |
| **Application/Software Architect** | Specialist | Deep dive into a single system or app | Optimize the internal architecture of a Python API |

<Callout type="idea" title="Key takeaway">
The SA is **more technical** than the Enterprise Architect but **broader** than the Application Architect. That's why the SAA exam covers so many services — it reflects the real breadth of the role.
</Callout>

## Who Should Take the SAA?

Regardless of your future specialization, the SAA gives you a solid foundation:

| Your area of interest | Is the SAA useful? | Reason |
|---|---|---|
| DevOps / SRE | ✅ Yes | Foundation for understanding the services you'll automate |
| Machine Learning | ✅ Yes | You need to understand the infrastructure where your models will run |
| Cybersecurity | ✅ Yes | 30% of the exam is security — it's the foundation for AWS Security Specialty |
| Development | ✅ Yes | You'll understand how the services your code consumes connect together |
| Management / Leadership | ✅ Yes | Gives you technical vocabulary to make informed decisions |

## Exam Format

| Aspect | Detail |
|---|---|
| Code | **SAA-C03** |
| Questions | 65 total (~130 minutes) |
| Style | Multi-service scenarios (not pure memorization) |
| Selection | Multiple choice (1 correct) or multiple response ("select 2") |
| Passing score | **720 / 1000** |

<Callout type="warn" title="Key difference from Cloud Practitioner">
The CCP asks **what** a service is. The SAA asks **how** to combine services to solve a scenario. Questions require **analysis and application**, not recall.
</Callout>

## Sample Exam Questions

These examples come directly from the course material and reflect the actual SAA-C03 question style.

### Example 1: NAT Gateway + Private Subnets

> A company runs a public-facing 3-tier web application in a VPC across multiple AZs. The application tier EC2 instances are in **private subnets** and need to download software patches from the internet. However, the instances **cannot be directly accessible from the internet**. What actions should be taken? *(Select 2)*

| Option | Correct? |
|---|---|
| A) Configure a NAT Gateway in a **public subnet** | ✅ |
| B) Define a custom route table with a route to the NAT Gateway and associate it with the private subnets | ✅ |
| C) Assign an Elastic IP to each instance | ❌ |
| D) Configure a NAT Instance in a **private subnet** | ❌ |
| E) Define a custom route table without associating it | ❌ |

<Callout type="success" title="Reasoning">
- **NAT Gateways must live in a public subnet** (otherwise they can't reach the internet to proxy traffic)
- Private subnets need a **route table pointing to the NAT Gateway** for internet-bound traffic
- An Elastic IP on the instance would expose it directly — violates the requirement
- A NAT Instance in a private subnet wouldn't work — it needs to be in a public subnet
</Callout>

### Example 2: EC2 Hibernation

> A solutions architect wants to save costs for EC2 instances that don't need to run during a 2-week company shutdown. The applications store data in **instance memory** that must be present when instances resume. What solution should be used?

| Option | Correct? |
|---|---|
| A) Take an EBS volume snapshot and restore | ❌ |
| B) Stop the instances and rely on EBS storage | ❌ |
| C) Run the applications with **hibernation enabled** and hibernate before shutdown | ✅ |
| D) Note instance types and relaunch with the same configuration | ❌ |

<Callout type="success" title="Reasoning">
**Hibernation** is the **only EC2 feature** that preserves instance memory (RAM) to disk. When resumed, the RAM is restored exactly as it was. Stopping an instance (stop) only preserves the EBS volume, not the RAM.
</Callout>

### Example 3: ENI Failover

> A monitoring application on EC2 uses its **private IPv4 address** to receive connections. Traffic needs to be quickly redirected to a standby instance if the primary fails. How should this be designed?

| Option | Correct? |
|---|---|
| A) Deploy an ALB and register both instances | ❌ |
| B) Configure DHCP to reassign the same IP | ❌ |
| C) Attach a **secondary ENI** with the private IP; move it to the standby if the primary fails | ✅ |
| D) Associate an Elastic IP and migrate it between instances | ❌ |

<Callout type="success" title="Reasoning">
- An **ALB** is for **active-active** configurations, not active-passive
- **DHCP** in AWS doesn't allow reassigning private IPs this way
- A **secondary ENI** with the private IP **can be moved** between instances — the correct solution for active-passive failover with a private IP
- An Elastic IP is public — the scenario uses a **private** IP
</Callout>

### Example 4: S3 Pre-signed URL

> An analytics company wants a webpage to execute a JavaScript script that makes **authenticated GET requests** to the company's S3 bucket. What should the architect do?

| Option | Analysis |
|---|---|
| Provide users with a **Pre-signed URL** for the script | ✅ — The pre-signed URL includes a temporary authentication token that enables the GET request without exposing credentials |
| Enable CORS on the bucket | Necessary but insufficient — doesn't solve authentication |
| Create a public IAM user | ❌ — Never expose IAM credentials in client-side JavaScript |

<Callout type="success" title="Reasoning">
A **Pre-signed URL** contains a token with an expiration date that authorizes the request without requiring the user to have AWS credentials. It's the standard mechanism for granting temporary, authenticated access to S3 objects.
</Callout>
